## 三、产品介绍文档结构

### 1. 项目一句话

TOMOAI OPC 是一套面向 AI 工具测评公众号的内容生产工作台，把 OPC 从“临时问 AI 写稿”升级为“可判断、可实测、可交付、可复盘”的内容运营系统。

### 2. 场景与痛点（Why）

AI 工具测评类公众号看起来只是写文章，实际工作链路很长：每天要从热点、产品更新、广告主需求和读者问题中筛选选题；判断是否值得写；把选题拆成可测试任务；完成真实上手、截图、成功案例和失败边界；再组织成公众号文章、标题、封面，并在发布前检查夸大表达、广告风险和 AI 味。

过去这个过程主要依赖人工经验和零散 AI 对话，存在几个明显问题：

- 选题判断不稳定：热点很多，但很多只是资讯，不能转成真实测评。
- 实测准备成本高：每篇文章都要重新列测试任务、截图清单、成功/失败案例。
- 商单容易失衡：广告主希望强调卖点，但测评号必须保留可信度和边界。
- 写作返工多：缺少稿前包时，文章容易跑题、空泛或变成 AI 套话。
- 发布风险难控：绝对化表达、硬广感、导流风险和证据不足常常到最后才发现。
- 经验难沉淀：一次次和 AI 对话得到的判断，很难变成下次可复用的流程。

因此，本项目选择了一个真实、具体、可验证的 OPC 场景：为 AI 工具测评公众号搭建一套完整内容生产工作台。

### 3. 解决方案（How）

TOMOAI OPC 将公众号内容生产拆成三个层次：

第一层是账号基础：明确账号定位、目标读者、适合测评的工具类型和作者表达风格，避免每次选题都从零开始。

第二层是内容生产：把热点、工具链接、广告主 Brief 等输入，转化为选题策划卡、商单分析、稿前包、文章框架、标题和封面提示词。

第三层是质量控制：在发布前检查标题正文一致性、夸大表达、AI 味、证据不足和导流风险；发布后再通过阅读、推荐、转发、留言和商单线索做数据复盘。

平台当前包含这些核心模块：

- 总览：集中展示今日任务、待处理选题、待写文章和风险提醒。
- 账号定位：根据主理人背景、经验、AI 使用基础和目标读者生成账号方向。
- 选题雷达：把热点和工具线索转成可判断的内容候选。
- AI 工具号日报：聚合近期 AI 热点，筛出适合账号的选题机会。
- 选题策划卡：判断读者痛点、内容切口、可测任务、商业潜力和风险。
- 商单 Brief：把广告主诉求转成可接受表达、必测任务和风险边界。
- 写文工作台：从确认选题生成稿前包、文章框架、正文草稿和标题备选。
- 封面头图：生成公众号封面与头图的文案、构图和提示词。
- 发布检查：检查夸大、硬广、AI 味、证据不足和风险表达。
- 数据复盘：用发布后表现反推下一轮选题和内容策略。

### 4. 小浣熊使用方式与关键 Prompt

本项目使用小浣熊的重点不是生成一次性文本，而是通过多轮交互完成三类工作：

- 需求拆解：把“做一个 AI 工具测评公众号平台”拆成账号定位、选题、稿前包、商单、写文、封面、发布检查和复盘模块。
- Prompt 沉淀：把每个模块背后的判断标准写成独立 Agent Markdown，成为后续可复用的工作说明。
- 前后端实现：在本地 Node 服务和单页工作台中实现模块调用、数据保存、预览、编辑和发布检查流程。

可在文档中展示的 Prompt 原词片段：

```text
你负责判断一条热点/工具线索是否适合当前 OPC 账号写。
输出必须是 JSON，不要输出 Markdown 代码块，不要解释。

判断规则：
- 纯资讯不能直接通过，必须能转成实测、教程、案例、对比、成本拆解、产品观察或商业判断。
- 优先考虑当前账号能长期持续输出、读者有真实痛点、能做出截图/案例证据的线索。
- 不适合账号方向的线索要明确降优先级或建议弃用。
```

```text
稿前包不是终稿，目标是帮主笔准备真实测评和文章骨架。
必须保留真实案例、截图清单、成功结果、失败边界、核心判断。
不直接硬吹工具，不夸大替代能力。
一篇文章只服务一个核心选题，不塞过多观点。
```

```text
检查公众号 Markdown 草稿是否存在发布前风险。
检查方向：
- 违禁词和敏感表达。
- 绝对化、夸大、硬广感。
- AI 味、套话、空泛总结。
- 标题和正文是否一致。
- 是否有真实案例、截图证据和失败边界。
- 是否存在明显导流风险。
```

这些 Prompt 的特点是：不是泛泛让 AI 写文案，而是把 OPC 工作中的判断标准结构化，并要求输出可被前端继续处理的 JSON 字段，从而形成可运行的工作流。

### 5. 模块协作方式

平台的核心流程是：

账号定位 -> 热点/工具线索 -> 选题判断 -> 选题策划卡 -> 商单 Brief 或普通稿前包 -> 文章框架 -> 正文草稿 -> 标题生成 -> 封面提示词 -> 发布检查 -> 发布后复盘

每个模块都保留人工确认环节。比如选题策划卡不会直接进入写作，必须先由用户确认；文章框架可编辑后再生成正文；发布检查只给出问题和修改建议，不替用户强制发布。这样既发挥 AI 的整理、判断和生成能力，也保留 OPC 对账号定位、商业边界和内容质量的最终控制。

### 6. 方案独特性

这个项目的创新点不在于“让 AI 写文章”，而在于把公众号内容生产中最难标准化的部分前置和结构化：

- 从资讯筛选转向实测任务：每个选题必须回答能不能测试、测什么、截图在哪里。
- 从单篇生成转向长期运营：账号定位、风格库、知识库和复盘模块让经验可以积累。
- 从硬广生成转向可信商单：商单 Brief 模块同时处理广告主诉求和测评边界。
- 从最终结果转向调优过程：选题、框架、稿前包、正文、标题和发布检查都可以多轮调整。
- 从聊天记录转向工作台资产：Prompt 被沉淀为 Agent 文件，数据被保存为本地工作区，下一次可以复用。

- 发布前风险检查与发布后复盘模块。
