# OPC 平台框架

## 产品定位

TOMOAI AI 工具测评 OPC 平台服务于一个明确场景：帮助 OPC 稳定生产 AI 工具相关公众号文章。

它不是通用聊天助手，也不是单纯的文章生成器，而是一个内容运营工作台：把账号定位、选题判断、工具实测、商单 Brief、稿前包、发布检查和数据复盘串起来。

## 核心用户

- 公众号主理人：决定账号方向、文章判断和商业边界。
- OPC：负责收集资料、整理 Brief、生成稿前包、辅助写文、发布前检查。
- 审稿人：检查观点、事实、合规和广告表达。

## 核心对象

- 工具：一个被测评的 AI 产品或功能。
- 选题：一个可写的文章角度，不等于一条新闻。
- 稿前包：写文前必须准备的结构化资料。
- 文章：公众号最终输出。
- 商单：带广告主要求、报价、交付节点和风险边界的合作项目。
- 规则：账号风格、平台合规、禁用表达和测评标准。

## 模块分层

### 第一层：账号基础

- AI 能力接入
- 账号定位
- 作者风格库
- IP 首篇文章

这一层解决“这个号是谁、写什么、怎么写、用什么能力辅助写”。

### 第二层：内容生产

- 选题雷达
- 商单 Brief
- 写文工作台
- 封面头图

这一层解决“下一篇写什么、凭什么写、怎么组织素材、交付什么内容”。

### 第三层：质量控制

- 发布检查
- 数据复盘

这一层解决“能不能发、发完表现如何、下一轮怎么改”。

## 当前迭代边界

当前先做框架，不做复杂工程化：

- 暂不引入前端框架。
- 暂不接真实 API。
- 暂不新增依赖。
- 暂不处理登录、权限、数据库。
- 先用静态 HTML + 文档 + JSON 模板承载结构。

后续当内容规则稳定后，再决定是否升级为真正的本地应用或 Web 应用。

## 判断平台是否成型

平台框架初步成型的标准：

- 每个模块知道自己接收什么资料。
- 每个模块知道自己产出什么结果。
- 每个模块知道什么结果算合格。
- 一篇 AI 工具测评文章能从选题一路走到发布检查。
- 新资料进来时，能明确放到工具库、选题池、商单库、风格库或规则库。
